What was learned

Lesson The lazy User import was cargo-culted

A # Late import to avoid circular dependency comment sat in dashboard.py for weeks. No circular dep existed — models.py never imports from dashboard.py, and views.py imports both. The pattern was copied from an earlier setup where the dep was real. Moving the import to the top of the file is safer and clearer.

Lesson related_name='+' breaks getattr-based access

The hospital scoping bug (get_hospital_from_user() always returning None) was caused by HospitalUserProfile.user = ForeignKey(related_name='+'). With related_name='+', Django doesn't create a reverse relation, so getattr(user, 'hospital_profile', None) silently returned None. Fix: direct query with select_related instead of trusting getattr.

Lesson SQLite FK constraints still block schema_editor.create_model()

Even after adding db_constraint=False to all FKs, Django's schema editor still failed during test database creation on SQLite. Root cause: SQLite has limited transactional DDL support — CREATE TABLE inside a transaction with FK enforcement raises NotSupportedError. Solution: raw CREATE TABLE IF NOT EXISTS with PRAGMA foreign_keys = OFF.

Lesson vi.clearAllMocks() vs vi.resetAllMocks()

Frontend tests using mockReturnValueOnce (for promise-based loading state) would infect subsequent tests. clearAllMocks resets call history but keeps return-value behavior. resetAllMocks also clears implementations. Switching fixed cross-test pollution in the Stats test suite.

Next move

Cover the highest-risk UI: NewHospitalWizard

The wizard is the most complex frontend component in the app — multi-step form with conditional fields, validation, and a POST to /hospitals/create/. It has zero test coverage. A single Vitest + RTL test that walks through all 3 steps and verifies the POST payload would catch regressions faster than any manual QA.

Second: add an AxiosError-aware toast/notification system so the frontend surfaces 500 errors instead of silently failing.

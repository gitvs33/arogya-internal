Duplicate platform_stats view

Files

  • views.py — platform_stats function (lines 237–267)
  • services/dashboard.py — _read_kpis() returns superset
  • frontend/src/pages/Stats.tsx — consumer of platform_stats
  • frontend/src/api/internalApi.ts — getStats() method

The numbers

  • platform_stats: 8 raw KPIs, 6 direct queries
  • read_overview(): 6 top-level keys including the same 8 KPIs
  • Leverage: 0 — the view duplicates query logic

Problem

The platform_stats view function makes 6 direct database queries (Hospital.objects.count(), UserLoginActivity.objects.filter(...), etc.) that produce a flat 8-key response. Exactly the same KPIs — plus module status, storage, license info, system info, and security — are already returned by dashboard_service.read_overview()['kpis']. The view is a shallower duplicate: its interface (flat list of 8 numbers) is nearly as complex as its implementation (6 queries).

Solution

Replace the 6 direct queries with a single delegation to dashboard_service.read_overview(). Extract the 'kpis' key and return it. The view becomes a 3-line adapter — query logic lives in one place (the service) instead of two.

Benefits

  • Locality: change the KPI definition in one place, not two
  • Test surface: tests for KPIs already exist in test_dashboard_service.py — no new test needed
  • Deletion test: delete the platform_stats function, the frontend is unchanged

Dashboard endpoints have no frontend consumer

Files

  • urls.py — 3 admin/dashboard/ paths
  • views.py — 3 adapter views (admin_overview, etc.)
  • services/dashboard.py — query logic
  • frontend/src/api/internalApi.ts — no dashboard methods

The numbers

  • Backend depth: 3 composite endpoints behind a service
  • Frontend consumption: 0 API calls
  • One-adapter seam: the service has exactly 1 adapter (the views)

Problem

The 3 composite dashboard endpoints (admin/dashboard/overview/, /activity/, /alerts/) exist only on the backend. The frontend has no code to call them, no React pages that render their data, and no internalApi.ts methods. Until a frontend consumer exists, the endpoints are dead code — the service seam isn't earning its keep.

Solution

Either build the dashboard frontend (pages + API methods) that consumes them, or remove the endpoints. A middle-ground: expose the service results through internalApi.ts so future frontend work has a contract to implement against.

Benefits

  • Leverage: the dashboard service's depth is only realised when a caller uses it
  • Deletion test: delete the 3 views + 3 URL patterns and nothing breaks
  • Two-adapter test: adding a frontend consumer makes the seam real
Worth exploring

Lazy User import at bottom of dashboard.py

Files

  • services/dashboard.py — line 189

The smell

  • User is used at lines 55, 60, 84, 87, 230, 243
  • But defined at line 189 via get_user_model()
  • Works because Python resolves at call time, not definition time

Problem

dashboard.py calls get_user_model() and assigns to User at the very bottom of the file (line 189). Functions above it (_read_kpis, _read_user_activity) reference User freely. In CPython this works because function bodies aren't evaluated until call time — but it's confusing, fragile, and triggers a NameError if any function is accidentally called at module load time.

Solution

Move the import to the top of the file. The circular import comment at line 188 says "Late import to avoid circular dependency" — but models.py doesn't import from dashboard.py, and views.py imports both models and dashboard (not the reverse). The circular dependency doesn't exist — the lazy import is cargo-culted.

Benefits

  • Clarity: User is defined before it's used
  • Safety: no runtime NameError risk from module-level evaluation
Worth exploring

No frontend tests

Files

  • frontend/ — no __tests__ dir
  • package.json — no test script
  • vite.config.ts — no test framework

The gap

  • 31 backend tests (dashboard service)
  • 0 frontend tests (pages, API client, auth)
  • Frontend pages have conditional rendering (isLoading, isError) that's never tested

Problem

The frontend is the primary consumer of every backend endpoint. The Stats.tsx page renders 6 stat cards from platform_stats — but there's no test that the 6 cards render correctly, that loading state shows skeletons, or that an error state shows a fallback. The response interceptor in client.ts redirects to /login on 401 — never tested.

Solution

Add Vitest (native to Vite ecosystem) with React Testing Library. Start with the simplest seam: the internalApi module's response types. Then add component tests for Stats.tsx (loading/error/success states) and the auth interceptor in client.ts.

Benefits

  • Locality: frontend bugs caught before hitting the browser
  • Leverage: the Stats page's data flow is fully covered in one place

StorageMetrics FK mismatch

Files

  • models.py — StorageMetrics.hospital field (line 290)

Problem

StorageMetrics.hospital = ForeignKey(Hospital, null=True, blank=True) without db_constraint=False. The FK references an unmanaged model (Hospital with managed = False). In test environments (SQLite), the FK constraint works because we create the table manually — but in a clean PostgreSQL test DB it could fail during syncdb. The null=True also allows orphaned records unassociated with any hospital.

Solution

Add db_constraint=False to the FK. This makes the constraint logical (enforced by Django) rather than physical (enforced by PostgreSQL), matching the pattern used by all unmanaged models' FKs to Hospital elsewhere in this codebase.

Speculative

Mixed GET/POST in hospital_list

Files

  • views.py — hospital_list function (lines 73–155)

Problem

The hospital_list view handles both GET (list hospitals) and POST (create hospital) in a single function with if request.method == 'GET': branching. The body is 82 lines with two very different responsibilities — listing is a simple annotated query, creating is a 10-step transactional workflow. This makes the function harder to read, test, and modify.

Solution

Split into two views: hospital_list (GET only) and hospital_create (POST only), each with its own URL pattern. The create logic could then be extracted into a HospitalOnboardingService module, making the transactional workflow independently testable.

Why speculative

The mixed method pattern is common in Django REST Framework and isn't inherently wrong. The deeper opportunity (extracting the onboarding workflow into a service) is worth exploring but depends on whether the onboarding logic is expected to grow.

Top recommendation

Fix the platform_stats duplication first

The platform_stats view is the clearest shallowness in the current codebase: its interface (a flat list of 8 numbers) is nearly as complex as its implementation (6 direct queries). The same KPIs already exist behind a service seam (dashboard_service.read_overview()['kpis']) with 6 passing tests. Delegating eliminates the duplication, shrinks the view to 3 lines, and makes the service seam earn its keep. It's a 5-minute change with zero risk.

Second: add a frontend consumer for the 3 dashboard endpoints (Candidate 2) — otherwise the service seam we created in the last refactor is still one-adapter speculation.

Dashboard API endpoint paths are wrong

Files

frontend/src/api/internalApi.ts

Problem

The three dashboard API methods use paths like /admin/dashboard/overview/ instead of /internal/admin/dashboard/overview/. With Axios base URL /api, the resulting HTTP request (/api/admin/dashboard/overview/) returns 404. The AdminDashboard page shows its error state and never displays data.

Solution

Add /internal prefix to all three dashboard endpoint paths.

Before / After

✗ Current

// internalApi.ts

get('/admin/dashboard/overview/')

→ /api/admin/dashboard/overview/

✗ 404 Not Found

✓ Fixed

// internalApi.ts

get('/internal/admin/dashboard/overview/')

→ /api/internal/admin/dashboard/overview/

✓ 200 OK

Why it's wrong: Every other API method in internalApi.ts uses the /internal/... prefix. These three don't — likely a copy-paste omission from when the old 11 flat endpoints were collapsed into 3 composite ones.

Query logic split between serializers and services

Files

backend/medos_internal/serializers.py backend/medos_internal/services/dashboard.py backend/medos_internal/views.py

Problem

Query logic is spread across three layers with no consistent seam. HospitalDetailSerializer.get_stats() queries Patient, Encounter, and Invoice — the same models that dashboard_service.read_platform_stats() queries. HospitalDetailSerializer.get_staff_breakdown() and get_admin() query HospitalUserProfile with role grouping. There is no single place to add or change hospital-related queries.

Solution

Extract a HospitalDetailService that provides read_stats(), read_staff_breakdown(), and read_admin(). The serializer becomes a pure data transformer — it receives pre-fetched data and formats it.

Deletion Test

If you delete the query methods from the serializers, the complexity doesn't vanish — it would need to move elsewhere. That means a real seam exists.

Current split (before)

Serializers bypass the service layer and query models directly, duplicating query logic.

✓ After: serializers receive data, don't query it

Admin user creation is inline business logic in a view

Files

backend/medos_internal/views.py

Problem

HospitalAdminViewSet.create() has ~30 lines of transactional business logic: creating a Django user, fetching a hospital, fetching or creating a role, creating a HospitalUserProfile. Error handling is a broad except Exception that catches everything. This is the last remaining inline business-logic-in-a-view since onboard_hospital() was extracted to a service.

Solution

Extract to a UserService.create_admin_user() in a new services/user_management.py module. The view becomes a thin 3-line adapter.

No pagination on hospitals list

Files

backend/medos_internal/views.py backend/medos_internal/serializers.py

Problem

hospital_list returns all hospitals with no limit. The serializer then runs N×2 additional queries — one for get_admin_email() and one for get_admin_name(), each doing a separate HospitalUserProfile lookup per hospital. With 50 hospitals that's 102 SQL queries for one request.

Solution

Add DRF PageNumberPagination (e.g. page_size=20). Consolidate the two admin queries into one prefetch in the view.

Current query pattern per hospital list request

Fix: Add Prefetch('staff_profiles__role') and compute admin email + name in Python. Drop the two SerializerMethodField queries.

Top recommendation

Fix the dashboard URL bug first — it's a production bug that makes the AdminDashboard page always show an error state. It's a one-line fix per endpoint (3 total) with zero risk.

Then extract HospitalDetail service — it addresses the deepest friction: query logic fragmented across layers. The deletion test passes: removing queries from serializers would not make complexity vanish. And it follows the established pattern (dashboard_service, hospital_onboarding_service, admin_crud).

The admin user creation extraction and pagination are both worthwhile but lower priority — the mutation at least works, and the N+2 query only hurts at scale.
